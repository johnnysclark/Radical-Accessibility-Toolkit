# Radical Accessibility — Potential Extensions & Roadmap

> Date: 2026-02-24
> Authors: John Clark, with Claude
> Context: Layout Jig (School Jig) CLI v2.3 is working with four recent extensions (TTS, History/Snapshots, Section Cut, MCP Server). Ethan's `fabric-accessible-graphics` repo (TACT + TASC) has been surveyed. This document maps the path from "working tool" to "fully inclusive architectural design environment for blind students."

## Part 1: Potential Next Extensions

### The Ten Core Extensions

These are ordered roughly by impact-to-effort ratio, with the most immediately valuable first.

---

#### 1. PIAF Vector Export (Direct from Layout Jig)

**Category**: Tactile fabrication
**Effort**: Medium
**Dependencies**: ReportLab (new, or borrow from TACT)

Layout Jig generates rich plans — walls, doors, hatches, Braille, legend — but can only output raster PNGs or 3D STLs. To get a PIAF print today, you'd screenshot and run it through TACT, which re-thresholds a raster image and loses all the vector precision.

A direct PDF export using ReportLab (borrowing TACT's `PIAFPDFGenerator` pattern) would write vector lines at exactly the right PIAF line weights (1.5mm grid, 2.5mm walls, 4mm columns), density-check the output, and embed BANA-standard Braille labels. Every line weight already in state.json (`heavy_lineweight_mm`, `wall_lineweight_mm`, etc.) maps directly to PIAF stroke widths.

**What changes**: New `cmd_piaf(state, tokens)` handler. New `piaf_export.py` companion module (like tactile_print.py). Reads state, builds vector geometry, writes PDF at 300 DPI. Density check warns if black coverage exceeds 45%.

**Why it matters**: Closes the design-to-tactile loop without any lossy intermediate step. Daniel designs, types `piaf export`, picks up the swell paper from the PIAF machine.

---

#### 2. Auto-Braille via Liblouis

**Category**: Accessibility infrastructure
**Effort**: Small
**Dependencies**: liblouis (optional — fallback ASCII map works without it)

Currently, `set bay A braille "⠠⠛⠁⠇⠇⠑⠗⠽"` requires the user to paste in pre-converted Unicode Braille. Daniel can't easily produce this himself.

When the user types `set bay A label "Gallery"`, the controller auto-populates `bay["braille"]` using Ethan's `BrailleConverter.convert_text()` — Grade 2 contracted Braille via liblouis, with the fallback ASCII-to-Braille map when liblouis isn't installed. Same for corridor labels, room labels, legend title, and block symbol labels.

**What changes**: One utility function `_auto_braille(text)` called after every label mutation. Liblouis import wrapped in try/except. Config field `"auto_braille": true` in state, toggleable via `braille auto on|off` and `braille grade 1|2`.

**Why it matters**: Removes the single most tedious manual step in the labeling workflow. Daniel types English, gets correct Braille automatically.

---

#### 3. Spatial Query Engine

**Category**: Accessibility infrastructure
**Effort**: Medium
**Dependencies**: None (stdlib math)

Daniel can't glance at the plan. He needs to interrogate spatial relationships programmatically:

- `query distance A B` → "Bay A to Bay B: 45 ft center-to-center, 20 ft edge-to-edge."
- `query adjacent A` → "Bay A shares edges with: Bay B (east), corridor (south)."
- `query path entry gallery` → "Entry → Corridor A → Bay C (Gallery): 3 rooms, 85 ft."
- `query rooms by area` → sorted list of all named rooms with square footage.
- `query sun south` → "South-facing rooms: Gallery (Bay A), Studio (Bay C). Total glazing: 3 windows, 42 linear ft."
- `query overlap` → "Bay A and Bay B overlap by 120 sq ft in region x=[40,60] y=[0,20]."

All computed from existing state data — bay origins, extents, corridor positions, aperture dimensions. Pure stdlib math, no new dependencies.

**What changes**: New `cmd_query(state, tokens)` with sub-handlers for each query type. Geometry utility functions for distance, intersection, adjacency, pathfinding.

**Why it matters**: Gives Daniel spatial awareness equivalent to a sighted person scanning the plan. The most frequent question a blind designer asks — "what's next to what?" — gets a precise, instant answer.

---

#### 4. Plan Image → Editable State (AI-Assisted Import)

**Category**: Image-to-model (novel)
**Effort**: Large
**Dependencies**: TACT OCR pipeline, Claude Vision API

The transformative extension. Use TACT's OCR pipeline (Tesseract positions + Claude Vision text) to read an *existing* floor plan image — a classmate's drawing, a textbook figure, a scanned precedent — and extract it into an editable `state.json`.

The AI identifies walls, rooms, dimensions, labels; the tool maps them to bays, corridors, apertures. The output isn't just a tactile image (what TACT does now) but a *parametric model* that Daniel can modify: "move the corridor 5 feet north," "add a door to the east wall."

**Workflow**:
1. `import image plan.jpg` → runs TACT analysis (OCR, dimension detection, grid overlay)
2. Claude Vision describes the plan structure (rooms, corridors, entries)
3. AI proposes a state.json with best-guess bays, corridors, walls
4. Daniel reviews via `describe`, corrects interactively ("set bay A spacing 25 20")
5. Result: a fully editable parametric model from a flat image

**What changes**: New `cmd_import(state, tokens)` plus an `import_plan.py` companion module. Uses TACT's processor + text detector + Claude API. Returns a candidate state that the user refines.

**Why it matters**: Every existing architectural drawing becomes accessible and editable. Daniel isn't limited to plans he creates from scratch — he can engage with the entire history of architecture, classmates' work, and textbook figures as manipulable models.

#### 5. Code and Dimensional Compliance Checker

**Category**: Design intelligence
**Effort**: Medium
**Dependencies**: None (stdlib)

`check ada` validates the plan against building code requirements:

- Corridor width >= 5 ft (ADA 403.5.1)
- Door clear width >= 32 in (ADA 404.2.3)
- Egress path from any room to exterior < 200 ft (IBC 1017.1 typical)
- Room area minimums by program type (classroom >= 600 sq ft, office >= 80 sq ft — configurable)
- Column-free span warnings (bay spacing > 40 ft for steel, > 25 ft for concrete)
- Dead-end corridor length < 20 ft (IBC 1020.4)
- Accessible route: at least one path with no steps and doors >= 32 in

Returns specific violations with repair commands:
```
ERROR: Corridor in Bay A is 4.0 ft wide (minimum 5.0 ft for ADA 403.5.1).
  Fix: corridor A width 5
WARNING: Bay B east span is 45 ft. Typical steel limit ~40 ft.
  Consider: set bay B spacing 20 20
OK: 3 of 5 checks passed. 1 error, 1 warning.
```

**What changes**: New `cmd_check(state, tokens)`. A `compliance.py` module with rule definitions (YAML or dict). Each rule is a function that takes state and returns pass/fail/warning with a fix suggestion.

**Why it matters**: Teaches building code *while designing*. Daniel learns ADA requirements through direct, contextual feedback rather than memorizing tables. Also catches errors that sighted students catch by visual inspection.

#### 6. Pegboard Wire Digitizer

**Category**: Physical-digital round-trip
**Effort**: Large
**Dependencies**: OpenCV (already in TACT's stack)

CLAUDE.md describes the pegboard: "0.065" rebar tie wire as lines. Overhead camera + OpenCV for wire position detection." This extension digitizes physical wire constructions into state.json.

**Workflow**:
1. Daniel bends wire on pegboard to sketch a wall layout
2. `capture` command triggers overhead camera frame capture
3. OpenCV detects wire positions (Hough line transform or contour detection)
4. Lines are classified as walls, mapped to bay coordinates
5. state.json updates with new wall segments
6. Rhino watcher rebuilds, TTS confirms: "OK: Captured 4 walls. Bay A east wall, Bay A south wall, Bay B north wall, corridor wall."
7. Daniel refines: "aperture A add d1 door x 1 5 3 7"

**What changes**: New `capture.py` module with camera interface and line detection. New `cmd_capture(state, tokens)` handler. Coordinate mapping from pegboard physical space to state.json model space (calibration step).

**Why it matters**: The only current input method is typing. The pegboard gives Daniel a physical sketching interface — bend wire, feel the result, capture to digital, refine. Physical-to-digital direction of the round-trip that CLAUDE.md calls for.

---

#### 7. Multi-Floor Stacking with Vertical Sections

**Category**: Design depth
**Effort**: Large
**Dependencies**: None (stdlib)

Architecture students think in section — how floors stack, where stairs punch through, how structure aligns vertically. Current tool is single-floor.

- `floor add 2` / `floor copy 1 to 2` / `floor set 2 active`
- Per-floor state (same bay/corridor/room structure, independent per floor)
- `section cut` extended to slice through stacked floors: wall-slab-wall-slab vertically
- `query align A floor1 floor2` → "Bay A columns align between floors 1 and 2: 12 of 12 match."
- `floor compare 1 2` → "Floor 2 differs from Floor 1: Bay C added, corridor B widened by 4 ft."
- STL export stacks floors with configurable slab thickness

**What changes**: State schema wraps current bays/corridors/rooms in `"floors": {"1": {...}, "2": {...}}`. An `"active_floor"` pointer determines which floor commands affect. All existing commands work unchanged on the active floor. Section cut and STL export gain multi-floor awareness.

**Why it matters**: Moves the tool from "floor plan generator" to "building designer." Section models that show vertical relationships are critical for understanding spatial sequences (double-height lobbies, split-level galleries, mezzanines).

#### 8. Audio Spatial Walkthrough (Sonification)

**Category**: Accessibility research
**Effort**: Large
**Dependencies**: winsound (stdlib on Windows) or pygame (optional)

An "explore" mode where Daniel navigates the plan as if walking through it:

- Arrow keys (or spoken directions) move a cursor through the plan
- At each step, TTS announces: current room name, dimensions, nearby elements
- Room size conveyed as reverb amount (small room = dry, large room = echo)
- Wall proximity as stereo panning (wall approaching on left = sound shifts left)
- Material changes as timbral shifts (concrete = low tone, glass = bright, wood = warm)
- Doorways as a "passage" chime, windows as a "light" tone
- Corridor transitions as a spatial narrowing sound

**What changes**: New `cmd_explore(state, tokens)` enters explore mode. `explore.py` module manages cursor position, collision detection, and audio rendering. Uses existing TTS for speech, plus tone generation for spatial cues.

**Why it matters**: Text description tells Daniel *what* the building is. Sonification tells him *what it feels like*. This is the research direction CLAUDE.md identifies — "binaural audio spatial walkthroughs." It's the closest a non-visual interface can get to the experience of walking through a space.

#### 9. Precedent Library (Canonical Buildings as State Files)

**Category**: Curriculum content
**Effort**: Medium (per building — scales with effort)
**Dependencies**: None

Pre-built state.json files for canonical buildings from the architecture curriculum:

- `precedent list` → "Available: farnsworth, salk, kimbell, villa_savoye, pantheon, barcelona, exeter_library, thermal_vals..."
- `precedent load farnsworth` → loads a curated state.json for Mies' Farnsworth House with accurate bay spacing, column grid, glass walls, terraces
- `precedent describe farnsworth` → returns an Arch-Alt-Text (Macro/Meso/Micro) description
- `precedent compare farnsworth kimbell` → "Both are single-story pavilions. Farnsworth: 1 bay, 8 columns, open plan. Kimbell: 6 bays, cycloid vaults, divided program."

Each precedent includes:
- `state.json` — editable parametric model
- `description.txt` — Arch-Alt-Text (Macro/Meso/Micro)
- `piaf.pdf` — pre-generated PIAF tactile card (once extension #1 exists)
- `source.txt` — dimensional data sources and citations

**What changes**: A `precedents/` folder with one subfolder per building. New `cmd_precedent(state, tokens)` handler that loads/describes/compares. Each building is a community contribution (John, Daniel, students, or AI-assisted).

**Why it matters**: Architecture education is built on precedent study. Sighted students flip through Archinect, Pinterest, monographs. Daniel gets a curated library of canonical buildings that he can explore, modify, and compare — in the same tool he uses to design.

#### 10. Rhino Viewport → Annotated PIAF (Screenshot Pipeline)

**Category**: Rhino integration + tactile fabrication
**Effort**: Medium
**Dependencies**: TACT processor module

Bridge the Rhino watcher's visual output back to tactile form:

1. Rhino watcher rebuilds geometry as usual
2. `export piaf from-rhino` triggers: Rhino captures viewport as high-res PNG (via RhinoScript `ViewCaptureToFile`)
3. Watcher writes the PNG path to `viewer_status.json`
4. Controller picks it up, pipes through TACT's processor: threshold, density check, Braille labels (auto-generated from state.json labels), PIAF line weight optimization
5. Outputs a PIAF-ready PDF of *everything visible in Rhino*

**What changes**: Watcher gains a `ViewCaptureToFile` command triggered by a flag in state.json. Controller gains `cmd_capture_rhino()` that sets the flag, waits for the image, and runs TACT processing. A `rhino_to_piaf.py` module handles the conversion pipeline.

**Why it matters**: Different from extension #1 because it captures the *rendered Rhino scene*, not just the state.json content. If a sighted TA has added annotations, sketched over the plan, or placed reference geometry in Rhino, it all appears in the PIAF print. This means sighted collaborators can draw freely in Rhino and Daniel gets tactile access to their additions.

### Additional Extension Ideas

Beyond the core ten, these are worth tracking:

#### 11. Template / Room Kit Library

Save and recall common room configurations:

- `template save bathroom_ada` → saves current cell configuration as reusable template
- `template load bathroom_ada into A cell 2,3` → applies the template
- Built-in kits: `ada_bathroom`, `classroom_30`, `office_private`, `mechanical_room`
- Each template carries dimensions, door placement, fixture zones, and compliance annotations

**Why**: Speeds up repetitive layout. A 30-student classroom always needs certain proportions and door placement.

#### 12. Guided Design Wizards

Step-by-step workflows for common building types:

- `wizard classroom_wing` → asks: "How many classrooms?" → "Corridor: single or double loaded?" → "Entry position: north, south, east, west?" → builds the layout incrementally
- `wizard house` → asks about bedrooms, bathrooms, kitchen position, garage, etc.
- Each wizard produces a valid state.json as a starting point for refinement

**Why**: Lowers the barrier to starting a design. Instead of "set bay A bays 4 3" (which requires knowing what you want), the wizard asks questions in architectural terms.

#### 13. Structural Heuristic Feedback

Real-time structural commentary as Daniel designs:

- "Bay A has a 45 ft clear span on the x-axis. For steel, consider adding an intermediate column or using a truss."
- "Corridor in Bay B has columns at 20 ft spacing — standard for flat slab construction."
- "Bay C cantilevers 12 ft beyond the last column line. Check deflection."

Not a structural analysis engine — just heuristic rules of thumb (span-to-depth ratios, typical material limits) that teach structural thinking while designing.

#### 14. Dimension Annotation Layer

Automatic dimensioning on the plan:

- `dimension on` → adds dimension strings to all bay spans, corridor widths, room areas
- `dimension style chain|aligned|running` → different dimensioning conventions
- Dimensions auto-update when geometry changes
- Braille dimension labels alongside English

**Why**: Sighted students read dimensions off the drawing. Daniel currently has to `describe` to get dimensions. Auto-dimensioning puts the numbers *on* the plan for PIAF output.

#### 15. DXF/DWG Export for Collaboration

Industry-standard CAD format export:

- `export dxf output.dxf` → writes a clean DXF with layers matching Layout Jig conventions
- Sighted classmates can open in AutoCAD, Revit, or Rhino and continue working
- Layer mapping: `JIG::Columns` → `A-COLS`, `JIG::Walls` → `A-WALL`, etc. (AIA layer standard)

**Why**: Daniel's work needs to integrate with the rest of the studio. DXF is the universal interchange format.

#### 16. Collaborative State Sync

Two users working on the same model simultaneously:

- Daniel in the CLI, sighted peer in Rhino (or a second CLI)
- Lock-free: each mutation writes to state.json with a sequence number
- Conflict resolution: last-write-wins for the same field, merge for independent bays
- Change notifications: "Bay B was modified by [other user]. Run 'describe B' to see changes."

**Why**: Architecture is collaborative. Daniel and his studio partner need to design together, not in serial.

#### 17. Voice Command Mode

Beyond TTS output — accept voice input:

- `voice on` enters voice mode
- Windows Speech Recognition (SAPI) or Whisper API transcribes spoken commands
- Fuzzy matching: "set bay alpha rotation thirty" → `set bay A rotation 30`
- Confirmation: "I heard: set bay A rotation 30. OK?" → "yes" → executes

**Why**: Daniel uses a braille display and keyboard, but voice is faster for some workflows, especially when hands are on the pegboard or holding a physical model.

#### 18. Feedback Annotation System

Capture design critique as structured data:

- `feedback add A "consider more natural light on the south facade"` → stores annotation linked to Bay A
- `feedback list` → shows all annotations with bay/room references
- `feedback from john "the corridor feels too narrow — compare to Kahn's Salk Institute"` → attributed feedback
- Export: annotations appear in PIAF print as numbered callouts with a key page

**Why**: Pin-up critiques are verbal and visual. This captures them in the model so Daniel can review and address them later.

#### 19. Acoustic Simulation Preview

Basic room acoustics estimation:

- `acoustic A` → "Bay A: volume 12,000 cu ft. RT60 estimate: 1.8s (reverberant). Consider absorption on ceiling."
- Uses Sabine equation with material assumptions (concrete = 0.02, glass = 0.04, acoustic tile = 0.70)
- Warns when rooms will be too live or too dead for their program type

**Why**: Sound is something Daniel *experiences*. Acoustic feedback connects the abstract design to a sensory reality he knows.

#### 20. Curriculum Assignment Templates

Pre-structured assignments that match John's studio curriculum:

- `assignment load week3_parti` → sets up site, constraints, and deliverables for Week 3
- `assignment check` → validates that the current design meets the assignment requirements (minimum room count, required program types, site coverage ratio)
- `assignment submit` → exports all deliverables (state.json, PIAF PDF, STL, description text, section SVG) to a named folder

**Why**: Standardizes the workflow from assignment to deliverable. Daniel doesn't have to remember which exports are needed — the assignment template defines them.

## Part 2: Roadmap to a Fully Inclusive Architectural Design CLI

This roadmap defines four phases, each building on the last. The goal: a blind architecture student can do everything a sighted student can do, without compromise, using text and tactile interfaces.

### Phase 0: Current State (Complete)

**What exists today** (Layout Jig v2.3 + recent extensions):

- Interactive REPL with ~40 commands for plan generation
- Rectangular and radial structural grids with irregular spacing
- Walls with door, window, and portal apertures (swing, hinge, sill height)
- Rooms with image-based hatch fills for tactile differentiation
- Corridors with centerline, loading zones, and dashed representation
- Braille + English labels with legend system
- Block symbols for PIAF-optimized aperture representation
- Tactile 3D: wall extrusion, section cut at configurable height, STL export
- Bambu Lab 3D printer integration (export → slice → FTP upload → MQTT print → status)
- Section cut export to SVG
- TTS audio feedback (PowerShell SAPI)
- Numbered history + named snapshots
- MCP server for AI assistant integration
- File-watcher architecture: CLI → state.json → Rhino watcher → geometry

**What exists in Ethan's repo** (available for integration):

- TACT image-to-PIAF pipeline (10 presets, density management, 300 DPI PDF)
- Braille conversion via liblouis (Grade 1 + 2, overlap detection, abbreviation keys)
- Three-engine OCR (Tesseract + EasyOCR + hybrid Claude Vision)
- RainbowTact color-to-tactile patterns
- Arch-Alt-Text description framework
- Tactile hatch material standards (6 patterns)
- PAI skills and hooks/learning system

### Architectural Principles Across All Phases

These principles from CLAUDE.md hold throughout:

1. **Text is the universal interface.** Every capability must work through typed (or spoken) commands with text (or spoken) responses. No visual-only features, ever.

2. **The CMA stays canonical.** `state.json` is the single source of truth. Every new feature reads from and writes to it. New transport layers (MCP, voice, pegboard) all produce valid CMA mutations.

3. **Zero-dependency core.** The controller CLI remains stdlib-only. New capabilities that need external libraries (ReportLab, OpenCV, liblouis) live in companion modules with graceful degradation when dependencies are missing.

4. **Crash-only viewer.** Rhino (and any future viewer) can crash at any time. The model survives on disk. This property must hold as we add floors, annotations, and collaborative features.

5. **Atomic writes everywhere.** Every file mutation goes through `_atomic_write()`. History, snapshots, exports, captures — all safe from half-written files.

6. **Screen-reader-first output.** `OK:` / `ERROR:` prefix protocol. One item per line. No tables, no ASCII art, no color-only indicators. Every new command follows this convention.

7. **Accessibility is tested, not assumed.** Every extension is tested with a screen reader (or flagged as untested). Daniel is a co-designer, not a test subject.

8. **Semantic over geometric.** Describe *what things are*, not just where they are. "The gallery is adjacent to the corridor on the south" beats "Rectangle at (40,0) to (80,60) shares edge with rectangle at (40,-8) to (80,0)."

### Success Criteria

The roadmap is complete when:

- A blind student can complete an entire architecture studio project — from parti sketch to final presentation — using only the CLI, tactile prints, and audio feedback
- The tool produces deliverables (plans, sections, models, descriptions) that are indistinguishable in quality from those produced with visual software
- Every canonical building in the first-year curriculum exists as a loadable, editable precedent
- The workflow has been validated by at least one blind architecture student through a full semester
- At least one publication documents the tool and its pedagogical impact

---

### What We're Really Building

This isn't assistive technology. We're not making Rhino "accessible" — we're building a parallel design environment where non-visual interaction is the primary case. The thesis of the Radical Accessibility project:

> By designing for blindness first, we create tools that are not only accessible but often superior to their visual-centric counterparts.

The spatial query engine is useful for *any* designer, not just blind ones. Compliance checking catches errors that sighted students miss. Precedent libraries with structured descriptions are better study tools than flipping through monographs. Audio walkthroughs convey spatial qualities that plan drawings can't.

The roadmap isn't about accommodation. It's about building the tools that architecture should have had all along.
